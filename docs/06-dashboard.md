# Dashboard

Dashboard — это веб-интерфейс, в котором видно всё, что бот делает.
По умолчанию доступен на **http://127.0.0.1:8080**.

## Как открыть

Пока бот запущен (`python main.py`), просто откройте в браузере:

```
http://127.0.0.1:8080
```

Если запустили бота на другой машине (например, на VPS) — замените
`127.0.0.1` на IP-адрес той машины. И обязательно **поставьте пароль**
в `.env` через `DASHBOARD_PASSWORD=…` — иначе ваши данные увидит
любой, кто знает адрес.

## Что за вкладки

Сверху страницы — несколько разделов.

### Overview — главный экран

- **Текущий баланс** — сколько денег на счёте (или в paper-кошельке).
- **PnL за день / неделю / месяц** — сколько заработали/потеряли.
- **Открытые позиции** — сколько и на каких символах.
- **Режим торговли** — paper или live. Live выделяется цветом, чтобы
  вы случайно не решили, что торгуете бумагой.
- **Последние сделки** — таблица с временем, символом, результатом.
- **Статус бота** — работает ли, когда последний heartbeat, нет ли
  сработавших circuit breakers.

Это первое, что стоит смотреть каждое утро.

### Positions — позиции

Все **открытые** позиции с подробностями:

- Символ, направление (long / short), размер.
- Цена входа, текущая цена, текущий PnL.
- Стоп-лосс и тейк-профит — где они стоят.
- Trailing-stop — если подтянулся, видно новый уровень.
- Кнопка **«Закрыть вручную»** (защищена PIN-кодом если задан).

### Trades — история сделок

Все **закрытые** сделки, с фильтрами по дате, символу, стратегии,
результату (win / loss). По каждой строке можно кликнуть и
увидеть:

- Какая стратегия дала сигнал.
- Какое было значение ML-модели (если она работала).
- Какие индикаторы на момент входа.
- Почему сделка закрылась — по стопу, тейку, trailing-у, тайм-стопу?

Это главный источник для **аудита**: «почему бот купил в 14:00 и
продал в 16:00 с минусом?». Тут вы найдёте ответ.

### Signals — сигналы

Все сигналы, которые стратегии сгенерировали — **даже
отклонённые**. По каждому видно:

- Какая стратегия.
- Прошёл ли multi-timeframe gate.
- Что сказала ML-модель.
- Через какие риск-гейты прошёл / на каком остановился.

Если бот «ничего не делает» — скорее всего, дело в том, что сигналы
есть, но они фильтруются. Тут это видно.

### Risk — риск-контроль

- Сколько сегодня потратили / потеряли.
- Сколько позиций открыто / сколько максимум разрешено.
- Состояние всех **circuit breakers** — какие зелёные (норма),
  какие горят оранжевым (предупреждение), какие красные
  (торговля заморожена).
- Последние срабатывания.

### ML — машинное обучение

- **Текущая модель**: когда обучена, на скольких сделках, какой
  stage (production / staging / archived).
- **Метрики**: precision, recall, AUC, F1 — каждая с доверительным
  интервалом от bootstrap.
- **Per-regime**: как модель работает на разных режимах рынка
  (trending_up, sideways, volatile). Если где-то откровенно плохо —
  это видно.
- **Walk-forward stability**: насколько стабильны метрики на
  разных временных фолдах (с учётом purge/embargo).
- **Confusion matrix**: сколько TP / FP / TN / FN на тестовом
  наборе.
- **Feature importance**: какие 10 признаков сильнее всего влияют
  на решения.
- **Shadow-mode vs live**: если модель в shadow-mode — видно, что
  она **бы** заблокировала, но не заблокировала (потому что в тени).
- **Champion vs challenger**: сравнение champion-модели и
  challenger-а в A/B-тесте.

Подробнее про ML → [05-машинное-обучение](05-машинное-обучение.md).

### ML-robustness — диагностика надёжности модели

Отдельная страница `/ml-robustness` — для тех, кто хочет понять,
**не случайно ли модель показывает хорошие метрики**:

- **PSR / DSR gate** — pass/fail бейджи. PSR — вероятность того,
  что истинный Sharpe больше бенчмарка; DSR — то же с поправкой
  на `N попыток`.
- **Feature-importance stability** — bar-chart важности признаков
  с error-bars (разброс между walk-forward фолдами) и
  Spearman-корреляция рангов.
- **Walk-forward diagnostics** — размеры train/test-окон,
  purge-ratio, метрики по каждому фолду отдельно.
- **Meta-label precision** — качество вторичного (meta-label)
  классификатора.
- **PSI drift** — текущий индекс дрейфа фичей.

### Observability — поток событий и здоровье системы

Страница `/observability`:

- **Поток событий в реальном времени** — падения, срабатывания
  circuit breaker-ов, PSI-дрейф, переобучения ML.
- **Текущий HMM-режим** и таймлайн переходов (bearish / neutral /
  bullish).
- **SLO burn** — насколько быстро «сгорает» бюджет ошибок.
- **Здоровье сервисов**: API Binance, БД, log-shipper, heartbeat
  watchdog-а.

### Strategies — стратегии

По каждой стратегии (EMA, Bollinger, MACD div, Mean Rev, Grid,
DCA):

- Включена / выключена.
- Сколько сигналов дала за период.
- Сколько пропустил multi-timeframe gate.
- Сколько пропустила ML-модель.
- Итоговый win-rate.

Эта вкладка отвечает на вопрос «какая стратегия работает лучше, а
какую стоит отключить».

### Events — события

Журнал всего, что произошло с ботом: запуски, остановки,
срабатывания circuit breakers, ошибки API, heartbeat-ы
watchdog-а, обучения ML-модели, etc.

Каждая строка — с временем, уровнем (INFO / WARN / ERROR),
категорией и сообщением. Фильтр по уровню — если интересуют только
ошибки, включаем **ERROR**.

### Settings — настройки (только просмотр)

Текущее содержимое `.env`, без секретов (API-ключи замаскированы).
Отсюда можно **только смотреть**. Менять — только через
редактирование `.env` и перезапуск бота. Так сделано специально:
dashboard не должен быть способом случайно изменить риск-лимиты.

## Мобильная версия

Dashboard адаптивный, в браузере телефона всё работает. Полноценный
просмотр, без урезанной версии.

## Если dashboard не открывается

1. **Бот запущен?** Проверьте, что `python main.py` реально
   работает в терминале, не упал.
2. **Правильный порт?** По умолчанию 8080. Если занят — смените
   `DASHBOARD_PORT` в `.env`.
3. **Правильный адрес?** Если бот на другой машине — нужен её IP,
   а не `127.0.0.1`.
4. **Firewall?** На VPS, возможно, порт закрыт снаружи. Откройте
   в правилах.
5. **Cookie / кэш?** Если после обновления бот стал «странно себя
   вести» — очистите cookie для адреса и перезалогиньтесь.

## Логи-источник

Dashboard **ничего не считает сам**. Всё, что он показывает,
берётся из:

- `sentinel/data/sentinel.db` — SQLite-база со сделками, сигналами,
  метриками.
- `sentinel/logs/events.jsonl` — журнал событий в JSON-строках.

То есть если dashboard что-то показывает «не то» — скорее всего,
проблема не в dashboard, а в том, что записалось в базу. Dashboard
— это окно, а не источник истины.
