## 1. EXECUTIVE SUMMARY

### Текущий Статус
Система имеет **хорошую архитектуру** (6 стратегий, ML фильтр, risk management, news sentiment, 20 features), но **серьёзные интиграционные ошибки** делают бóльшую часть этих возможностей **неактивной**.

### Главная Проблема
**~60% написанного кода НЕ РАБОТАЕТ в продакшене:**

| Компонент | Статус | Влияние |
|-----------|--------|---------|
| ML Predictor | `rollout_mode="off"` — ВЫКЛЮЧЕН | ML не фильтрует сигналы |
| Auto Strategy Selection | `False` по умолчанию | Работает ТОЛЬКО EMA Crossover |

| Kelly Criterion | Использует `win_rate=0.5, avg_win=3%, avg_loss=2%` — ЗАХАРДКОЖЕНО | Не адаптируется к реальным результатам |

### Ожидаемое Улучшение
При исправлении всех проблем ожидается **+30-50% относительного улучшения Win Rate** (например, с 48% до 65-70%).

## 2. КРИТИЧЕСКИЕ БАГИ — "Мёртвый Код"

### 🔴 БАГ #1: FORWARD-LOOKING BIAS в ML (САМЫЙ КРИТИЧЕСКИЙ)

**Файл**: `analyzer/ml_predictor.py` → `extract_features()`

**Проблема**: Функция извлечения features использует данные, которые НЕДОСТУПНЫ на момент принятия решения:

```python
# ❌ ИСПОЛЬЗУЕТ ДАННЫЕ ИЗ БУДУЩЕГО:
price_delta = trade.exit_price - trade.entry_price     # exit_price ещё не известен!
price_delta_pct = price_delta / entry_price
regime_bias = self._regime_bias(trade.market_regime, trade.pnl_pct)  # pnl_pct из будущего!
macd_hist = regime_bias * max(abs(trade.pnl_pct) / 100.0, ...)      # опять pnl_pct!
```

**Это значит**: Модель ОБУЧАЕТСЯ на ответах, которые она потом пытается предсказать. На бэктесте показывает ~80% accuracy, в реальности — монета (50%).

**Конкретные поля с forward-looking:**
- `trade.exit_price` — цена выхода (неизвестна при входе)
- `trade.pnl_pct` — итоговый P&L сделки (неизвестен при входе)
- `trade.max_drawdown_during_trade` — макс просадка ВО ВРЕМЯ сделки
- `trade.max_profit_during_trade` — макс прибыль ВО ВРЕМЯ сделки
- `trade.hold_duration_hours` — длительность удержания

**Решение**: `extract_features()` должен использовать ТОЛЬКО данные, доступные ДО входа в сделку:
```
✅ trade.rsi_at_entry
✅ trade.adx_at_entry
✅ trade.volume_ratio_at_entry
✅ trade.confidence (рассчитана ДО входа)
✅ trade.hour_of_day, trade.day_of_week
✅ trade.market_regime (определён ДО входа)
✅ trade.news_sentiment, trade.fear_greed_index
✅ Историческая статистика (win_rate, hours_since_last, consecutive_losses)
```

**Влияние исправления**: +15-25% к РЕАЛЬНОМУ Win Rate (убирает иллюзию хороших результатов)

## 3. ML PREDICTOR — Полный Разбор

### Текущая Архитектура
```
RandomForestClassifier (200 деревьев, depth=8, balanced weights)
→ StandardScaler
→ TimeSeriesSplit CV (5 folds)
→ 80/20 train/test split
→ 20 features
```

### Проблема 3.1: Feature Engineering — Forward-Looking (см. БАГ #1)

5 из 20 features используют данные из будущего. Это **инвалидирует всю модель**.

### Проблема 3.2: Overfitting Gate Слишком Слабый

```python
if overfit_gap > 0.15:  # 15% разрыв допускается!
    logger.warning("ML OVERFITTING detected...")
    # НО МОДЕЛЬ ВСЁ РАВНО СОХРАНЯЕТСЯ!
```

**Проблемы:**
1. 15% gap — это ОГРОМНЫЙ разрыв. Профессиональный стандарт: **< 5%**
2. При обнаружении overfitting модель всё равно сохраняется (только warning)
3. Нет механизма отката к предыдущей рабочей модели

**Решение:**
```
gap > 5% → НЕ СОХРАНЯТЬ модель, откатить к предыдущей
gap > 10% → ALERT + отключить ML фильтр
```

### Проблема 3.3: min_skill_score = 0.60 — Слишком Низко

skill_score 0.60 означает модель лишь на 10% лучше случайного угадывания (0.5). Это **недостаточно** для фильтрации торговых сигналов.

**Профессиональный стандарт**: min_skill_score >= **0.72**

### Проблема 3.4: Block Threshold = 0.55 — Почти Монета

```python
block_threshold: float = 0.55
```

Модель блокирует сигналы только при уверенности < 55%. Это слишком мягко — модель пропускает почти всё.

**Решение**: block_threshold >= **0.60**, постепенно поднимать до 0.65

### Проблема 3.5: Нет Feature Importance Анализа

Модель обучена на 20 features, но нет информации о том, какие features полезны, а какие — шум. Random Forest имеет встроенный `feature_importances_`, но он НЕ логируется и НЕ используется.

**Решение**: После обучения логировать top-10 features by importance, удалять features с importance < 1%

## 4. СТРАТЕГИИ — Проблемы Каждой

### 4.1 EMA Crossover RSI (единственная активная!)

**Проблемы:**

#### a) Нет минимального порога пересечения
```python
is_crossover = prev_diff <= 0 and current_diff > 0
```
Любое пересечение EMA9/EMA21 генерирует сигнал, даже на 0.01%. В боковом рынке это создаёт **whipsaw** — множество ложных сигналов.

**Решение**: Требовать `current_diff > entry_price * 0.001` (0.1% минимальный спред)

#### b) Confidence из коррелированных факторов
```
base=0.50 + RSI<50(+0.10) + Volume>2x(+0.10) + close>EMA50(+0.10) + MACD>0(+0.10) + ADX>25(+0.05)
```
RSI, MACD, EMA — все трендовые индикаторы. Когда тренд есть — ВСЕ дают бонус. Когда нет — ВСЕ штрафуют. Это **inflation** confidence в хороший момент и **deflation** в плохой.

**Решение**: Использовать РАЗНОРОДНЫЕ факторы: volume (рыночная структура) + sentiment (внешний) + volatility (волатильность) + trend (технический). Максимум 1 технический индикатор для confidence.

#### c) Фиксированные SL/TP (3%/5%)
```python
stop_loss_pct: float = 3.0
take_profit_pct: float = 5.0
```
Не адаптируются к волатильности. В тихий рынок 3% SL слишком широк, в волатильный — слишком узок.

**Примечание**: `dynamic_sltp.py` написан, но НЕ ИСПОЛЬЗУЕТСЯ в стратегиях! Стратегии считают SL/TP сами.

---

### 4.2 Bollinger Breakout

**Проблемы:**

#### a) Squeeze Detection Сломан
```python
squeeze_threshold: float = 0.05  # это 5% BB bandwidth
```
BB bandwidth = (upper - lower) / middle. Значение 0.05 (5%) — это **абсолютный** порог, но bandwidth зависит от инструмента и волатильности. Для BTC нормальный bandwidth может быть 3-8%.

**Решение**: Использовать **percentile** — squeeze = bandwidth < 20-й перцентиль за 100 свечей

#### b) Нет подтверждения после breakout
Стратегия покупает при close > upper BB, но не проверяет, УДЕРЖАЛСЯ ли breakout. Часто цена пробивает BB и тут же возвращается (false breakout).

**Решение**: Требовать 2-3 свечи подряд над BB, или close > upper BB И volume > 2x

---

### 4.3 Mean Reversion

**Проблемы:**

#### a) "Не ловит нож" — Но пытается
```python
# BUY: RSI < 25 + close < lower BB
```
В даунтренде lower BB **сам движется вниз**. Цена может быть ниже BB И продолжать падать. RSI < 25 в сильном даунтренде — это не "перепроданность", а "тренд".

**Решение**: Добавить ОБЯЗАТЕЛЬНЫЙ фильтр тренда:
- НЕ покупать если `ema_9 < ema_21 < ema_50` (чётко даунтренд)
- Требовать ADX < 25 (отсутствие сильного тренда) ИЛИ бычью дивергенцию RSI

#### b) Слишком быстрый выход
```python
if features.close >= features.ema_21 > 0:
    return SELL  # revert to EMA21
```
Выход при достижении EMA21 — это захват лишь ~30-40% потенциального движения. Mean reversion может доходить до EMA50 и выше.

**Решение**: Не выходить полностью при EMA21, а ставить trailing stop с активацией при достижении EMA21.

---

### 4.4 MACD Divergence

**Проблемы:**

#### a) Lookback только 5 свечей
```python
min_divergence_bars: int = 5
```
Дивергенция за 5 свечей (5 часов на 1h) — это шум. Настоящие дивергенции формируются за **15-30+ свечей**.

**Решение**: `min_divergence_bars = 10`, `lookback_candles = 50`

#### b) Нет измерения силы дивергенции
Код проверяет только факт (есть/нет), но не силу. Сильная дивергенция (цена -5%, MACD +20%) намного надёжнее слабой (цена -0.1%, MACD +0.1%).

**Решение**: Вычислять `divergence_strength = |price_change| + |macd_change|` и использовать как множитель confidence.

---

### 4.5 DCA Bot

Текущая реализация — простой DCA (покупать регулярно). Нет адаптации к рыночным условиям.

**Улучшение**: Conditional DCA — покупать больше при RSI < 30, меньше при RSI > 70.

---

### 4.6 Grid Trading

Текущая реализация — базовая сетка. Нет динамической подстройки уровней.

**Улучшение**: Spacing пропорционально ATR, а не фиксированное.

Файл `risk/dynamic_sltp.py` содержит отличную ATR-based систему SL/TP с настройками per-strategy. **Но стратегии используют свои фиксированные значения!**

**Решение**: Стратегии должны вызывать `calculate_dynamic_sltp()` вместо hardcoded SL/TP.

### 5.3 Нет Cooldown После Серии Убытков

```python
# Текущее поведение: после 3 consecutive losses...

```

**Решение**: После 3 убытков подряд → cooldown 2 часа. После 5 → cooldown 6 часов.

## 6. MARKET REGIME — Детекция Рынка

### 6.1 Слишком Простая Логика

```python
if atr_ratio > 0.04:                          → VOLATILE
elif ema_9 > ema_21 > ema_50 and adx > 25:   → TRENDING_UP
elif ema_9 < ema_21 < ema_50 and adx > 25:   → TRENDING_DOWN
elif adx < 20 and close between BB:           → SIDEWAYS
else:                                          → UNKNOWN
```

**Проблемы:**
1. **Нет гистерезиса** — режим может переключаться КАЖДЫЙ цикл. Если ADX колеблется вокруг 25, режим будет прыгать trending ↔ sideways каждый час.
2. **ATR > 0.04 слишком грубо** — для BTC при $60k это $2400 ATR. Для $20k это $800. Один порог не работает для разных цен.
3. **UNKNOWN покрывает ~30-40% времени** — если EMA не идеально выстроены И ADX > 20, получаем UNKNOWN. Стратегии не знают что делать.

**Решение:**
1. Гистерезис: для выхода из режима нужно N свечей подряд другого сигнала
2. ATR в процентах от цены (уже делается через `atr_ratio`), но пороги нужно калибровать
3. Добавить промежуточные состояния: WEAKLY_TRENDING, TRANSITIONING

## 7. БЭКТЕСТИНГ — Проблемы Валидации

### 7.1 Walk-Forward ЕСТь, Но Не Используется По Умолчанию

`engine.py` имеет `run_walk_forward()`, но стандартный `run()` делает простой прогон без out-of-sample тестирования.

### 7.2 Slippage/Commission Модель Примитивная

```python
commission_pct: float = 0.1    # фиксированная 0.1%
slippage_pct: float = 0.05     # фиксированный 0.05%
```

**В реальности:**
- Slippage зависит от ликвидности и размера ордера
- При сильном движении slippage может быть 0.5-1%
- Commission зависит от объёма торгов (maker/taker tiers)

**Решение**: Slippage = max(0.05%, ATR * 0.1) — больше при высокой волатильности

### 7.3 Safety Discount = 0.70

```python
safety_discount: float = 0.7  # 70% от бэктест результатов
```

Это хороший подход, но 70% — оптимистично. Профессиональный стандарт: **50-60%**.

## КРИТИЧЕСКИЙ ВЫВОД

> **Самый быстрый способ поднять Win Rate — НЕ ПИСАТЬ НОВЫЙ КОД, а ПОДКЛЮЧИТЬ существующий.**

Фазы 1+2 дадут **+25-40%** улучшения Win Rate при минимальных изменениях кода. Основная проблема системы — не алгоритмы, а **интеграция**. Модули написаны хорошо, но не соединены друг с другом.

### Аналогия Профессионального Трейдера:

Представь, что у тебя:
- Отличный торговый терминал (есть) 
- 6 стратегий с чёткими правилами (есть)
- AI-фильтр сделок (есть, но ВЫКЛЮЧЕН)
- Динамический размер позиции (есть, но не подключен)
- Мультитаймфреймный анализ (есть, но данные не поступают)

Ты торгуешь ОДНОЙ стратегией, с фиксированным SL/TP, без фильтра, без дневного графика. Это как ехать на Ferrari на первой передаче с завязанными глазами — мощность есть, но она не используется.

**Приоритет #1: Развязать глаза (подключить daily TF)**
